# 进化机制与复盘框架

> 本文档定义项目如何持续进化、自我纠错、保持生命力。

---

## 一、进化循环

### 1.1 四层时间循环

```
日循环（Daily Loop）
├── 采集 → 筛选 → 验证 → 产出 → 更新
├── 频率：每日
├── 耗时：2-5小时（随自动化程度递减）
└── 输出：每日信号 + 验证报告

周循环（Weekly Loop）
├── 复盘 → 评估 → 调整 → 优化
├── 频率：每周日
├── 耗时：2小时
└── 输出：周复盘报告

月循环（Monthly Loop）
├── 全景更新 → 趋势分析 → 方向调整 → 目标校准
├── 频率：每月1号
├── 耗时：4小时
└── 输出：月度全景报告 + 趋势预判

季循环（Quarterly Loop）
├── 全量审计 → 方法论迭代 → 战略调整
├── 频率：每季度首日
├── 耗时：8小时
└── 输出：季度审计报告
```

### 1.2 进化触发器

**自动触发：**
- 某方向连续3天无新信号 → 标记为"停滞"
- 某方向可信度评分下降 → 触发重新验证
- 30天未更新的调研 → 触发复查提醒
- 预测准确率低于60% → 触发方法论审计

**手动触发：**
- 发现重大政策变化（如新法规、平台新规）
- 发现新的重大机会窗口
- 用户反馈指出错误
- 新工具/平台出现可能改变格局

## 二、预测准确率追踪

### 2.1 追踪机制

每条预测必须包含：
```json
{
  "prediction": "预测内容",
  "date": "YYYY-MM-DD",
  "timeframe": "30天|90天|180天",
  "confidence": "高|中|低",
  "verificationDate": null,
  "result": null,
  "reason": null
}
```

### 2.2 验证周期

| 预测类型 | 验证周期 | 验证方法 |
|----------|----------|----------|
| 收入预测 | 30天 | 对比实际案例数据 |
| 趋势预测 | 90天 | 对比市场实际变化 |
| 方向判断 | 180天 | 对比该方向实际发展 |
| 风险预警 | 30天 | 检查是否实际发生 |

### 2.3 准确率计算

```
准确率 = 验证正确的预测数 / 已到验证期的预测总数 × 100%

目标：
- 3个月：≥60%
- 6个月：≥70%
- 12个月：≥80%
```

### 2.4 偏差分析

当预测偏差>30%时，必须执行偏差分析：

```markdown
## 偏差分析

**预测：** [原预测内容]
**实际：** [实际情况]
**偏差：** [偏差百分比]

**原因分析：**
1. 信息不足（遗漏了什么信息）
2. 逻辑错误（推理链条哪里出了问题）
3. 外部变化（哪些不可预见的因素）

**改进措施：**
1. [具体改进动作]
2. [需要增加的信息源]

**更新后的预测：** [修正后的预测]
```

## 六、知识沉淀机制

### 6.1 从调研到知识库

每完成一次深度调研，自动沉淀以下知识：

```
调研文档 → 提取 → 知识条目
├── 工具知识：哪些工具好用、成本多少、学习曲线
├── 平台知识：各平台规则、变现机制、政策变化
├── 法律知识：相关法规、判例、红线
├── 方法论知识：什么验证方法有效、什么容易误判
└── 模式知识：成功模式、失败模式、常见陷阱
```

### 6.2 知识条目格式

```json
{
  "id": "knowledge-NNN",
  "type": "tool|platform|legal|method|pattern",
  "title": "知识标题",
  "content": "知识内容",
  "sources": ["来源1", "来源2"],
  "confidence": 4,
  "lastUpdated": "YYYY-MM-DD",
  "relatedResearch": ["research-001", "research-002"]
}
```

### 6.3 知识更新规则

- **自动更新**：当相关调研被更新时，关联知识条目自动标记为"需复查"
- **手动更新**：发现新信息时主动更新
- **过期淘汰**：超过90天未更新且无相关调研的知识条目标记为"待验证"
